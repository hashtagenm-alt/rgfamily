# ADR-004: VIP 클릭 = 시그니처 자격(11명) ≠ VIP 리워드(49명)

| 항목 | 값 |
|------|-----|
| 상태 | 승인 |
| 날짜 | 2026-02-03 |

## 문맥 (Context)

RG Family에는 두 가지 다른 VIP 개념이 혼재한다:
1. **VIP 리워드** - 시즌 랭킹 Top 49 후원자에게 부여되는 보상 (vip_rewards 테이블)
2. **시그니처 자격** - 당일 특정 하트 임계값을 달성한 후원자에게 부여되는 BJ 시그니처 클릭 권한 (11명 내외)

이 둘을 같은 "VIP"로 처리하면 49명 모두가 시그니처에 접근하게 되어, 시그니처의 희소성과 보상 의미가 사라진다.

## 결정 (Decision)

`signature_eligibility` 테이블과 `vip_clickable_profiles` View로 분리 관리한다:

- **시그니처 자격 판정**: 당일 후원 하트 임계값 기반 3단계
  - 10만 하트: 1단계 시그니처 접근
  - 15만 하트: 2단계
  - 20만 하트: 3단계 (풀 시그니처)
- **vip_clickable_profiles** View: 자격이 있는 프로필만 노출
- VIP 리워드(vip_rewards)는 별도 테이블로 독립 유지

## 결과 (Consequences)

**장점**:
- VIP 리워드와 시그니처 자격의 명확한 분리
- 임계값 조정이 DB 레벨에서 가능 (코드 변경 불필요)
- View를 통한 보안 (자격 없는 사용자에게 시그니처 비노출)

**단점**:
- 관리자가 두 시스템을 별도 관리해야 함
- 임계값 변경 시 기존 자격 데이터와의 정합성 확인 필요

## 관련 파일
- `supabase/migrations/20260203_signature_vip_click_system.sql` - 테이블/View 생성
- `src/app/vip/` - VIP 페이지 (리워드)
- `src/app/signatures/` - 시그니처 페이지 (자격 기반 접근)
